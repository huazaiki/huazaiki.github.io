    Introduction 介绍

    我们在编程调试过程中会遇到各种错误，Python调试器会打印如下的类似内容：

    Traceback (most recent call last):
  File "<pyshell#29>", line 3 in <module>
    result = buggy(5)
  File <pyshell#29>", line 5 in buggy
    return f + x
TypeError: unsupported operand type(s) for +: 'function' and 'int'

    这称为traceback_ message回溯消息。它将打印出导致错误的函数调用链，最新的函数调用位于底部。我们可以据此来找出导致问题的函数。

    Traceback Messages 回溯消息

    回溯消息会配对出现，此类配对信息中第一行具有以下形式：

    File "<file name>", line <number>, in <function>
    • File name: 包含问题的文件的名称。
    • Number: 文件中导致问题的行号，或包含下一个函数调用的行号
    • Function: 可以在其中找到行的函数的名称。

    第二行显示下一次函数调用的实际代码行（比第一行缩进一些），这会让我们快速查看到传递到函数中的参数以及上下文中使用的什么函数。

    Error Messages 错误消息

    回溯信息中的最后一行是error statement错误语句，格式如下：

    <error type>: <error message>
    • Error type: 导致的错误类型（例如 SyntaxError语法错误 ， TypeError类型错误 ）。这些通常具有足够的描述性，可帮助您缩小对错误原因的搜索范围。
    • Error message: 对导致错误的确切原因的更详细描述。不同的错误类型会产生不同的错误消息。

    Debugging Techniques 调试技术

    运行 doctests

    doctests是Python中一种快速为代码编写测试的好方法，格式如下：

    def foo(x):
    """A random function.

    >>> foo(4)
    4
    >>> foo(5)
    5
    """

    文档注释中看起来像解释器输出行的就是doctests。要运行这里面的内容，转到终端输入：

    python -m doctest file.py

    除了告诉你哪些文档测试失败了，它还会告诉你哪些文档测试通过了。

    编写自己的测试

    编写测试之前有一些小建议：

    • Write some tests before you write code: 这称为测试驱动开发。写下你期望函数首先表现的方式 – 这可以指导你编写实际代码。
    • Write more tests after you write code: 确定代码通过初始文档测试后，再编写一些测试来处理边缘情况。
    • Test edge cases: 确保代码适用于所有特殊情况。

    使用 print 语句

    一旦文档测试告诉你错误在哪里，你就必须弄清楚出了什么问题。如果 doctest 给了你一条回溯消息，请查看它是什么类型的错误，以帮助缩小搜索范围。还要检查您是否没有犯任何常见错误。

    我们第一次学习编程时，可能很难发现代码中的错误。一种常见的方法就是添加print语句。例如下面的foo函数不断地返回错误信息：

    def foo(x):
    result = some_function(x)
    return result // 5

    我们可以在返回之前添加一个 print 语句来检查 some_function 返回的内容

    def foo(x):
    result = some_function(x)
    print('DEBUG: result is', result)
    return other_function(result)

    注意：在调试语句前加上特定的字符串 DEBUG: 以便于我们更方便地查看调试信息

    如果事实证明 result 它不是我们期望的那样，我们会去 some_function 看看它是否正常工作。否则，我们可能必须在返回之前添加一个 print 语句来检查 other_function ：

    def foo(x):
    result = some_function(x)
    print('DEBUG: result is', result)
    tmp = other_function(result)
    print('DEBUG: other_function returns', tmp)
    return tmp

    一些建议：

    • 不要只是打印出一个变量 – 添加某种消息以使其更容易阅读：

      print(tmp)   # harder to keep track
print('DEBUG: tmp was this:', tmp)  # easier

    • 使用 print 语句查看函数调用的结果（即在函数调用之后）。

    • 在 while 循环结束时使用 print 语句查看每次迭代后计数器变量的状态：

      i = 0
while i < n:
    i += func(i)
    print('DEBUG: i is', i)

    • 不要只是在明显正确的行后面放随机 print 语句。

    长期调试

    上面描述的 print 语句用于快速调试一次性错误 – 找出错误后，您将删除所有 print 语句。

    但是，有时如果需要定期测试文件，我们希望保留调试代码。如果每次我们运行文件时都会弹出调试消息，这可能会有点烦人。避免这种情况的一种方法是使用全局 debug 变量：

    debug = True

def foo(n):
i = 0
while i < n:
    i += func(i)
    if debug:
        print('DEBUG: i is', i)

    现在，每当我们想做一些调试时，我们可以将全局 debug 变量设置为 True ，当我们不想看到任何调试输入时 ，我们可以将其转换为 False （这样的变量称为“标志”）。

    使用 assert 语句

    使用assert断言关键词来测试一个条件是否为真，否则会打印一行错误消息。这在我们知道某些点上的某些条件需要为真的时候真的很有用。

    例如，如果你正在编写一个函数，它接受一个整数并将其加倍，那么确保你的输入实际上是一个整数可能是有用的。然后可以编写以下代码：

    def double(x):
    assert isinstance(x, int), "The input to double(x) must be an integer"
    return 2 * x

    注意这里，我们并没有真正的调试double函数，我们所做的是确保调用double函数的任何人都要使用正确参数。例如我们有一个函数g，他接受一个字符串和一个数字，并将字符串的长度加到数字的两倍，实现如下：

    def g(x, y):
    return double(x) + y # should be double(y) + len(x)

    我们得到的不是一个奇怪的错误，即不能将字符串和数字相加，而是一个干净的错误，即 double 的参数必须是整数。这使我们能够快速缩小问题范围。

    assert语句的一个主要好处是，它们不仅仅是一个调试工具，您可以将它们永久保留在代码中。软件开发中的一个关键原则是，通常情况下，代码崩溃比产生错误的结果更好，并且在代码中使用断言会使代码更有可能在有bug的情况下崩溃。

    Error Types 错误类型

    SyntaxError

    • Cause: 代码语法错误

    • Example: 

        File "file name", line number
    def incorrect(f)
                    ^
SyntaxError: invalid syntax

    • Solution: ^ 符号指向包含无效语法的代码。错误消息不会告诉你什么是错的，但它会告诉你哪里错了。

    • Notes: Python在执行任何代码之前都会检查 SyntaxErrors 。这与其他错误不同，其他错误仅在运行时引发。

    IndentationError

    • Cause: 缩进不当

    • Example:

        File "file name", line number
    print('improper indentation')
IndentationError: unindent does not match any outer indentation level

    • Solution: 显示缩进不正确的行。简单地重新缩进这一行。

    • Notes: 如果你不同意制表符和空格，Python会抛出其中一个。确保使用空格！(It只是在Python中使用空格一般不那么令人头痛，所有CS61A内容都使用空格）。

    TypeError

    • Cause 1: 

      • 原始运算符的操作数类型无效。您可能正在尝试添加/减/乘/除不兼容的类型。

      • Example: 

        TypeError: unsupported operand type(s) for +: 'function' and 'int'

    • Cause 2: 

      • 在函数调用中使用非函数对象。

      • Example: 

        >>> square = 3
>>> square(3)
Traceback (most recent call last):
  ...
TypeError: 'int' object is not callable

    • Cause 3: 原因三：

      • 向函数传递错误数量的参数。

      • Example: 

        >>> add(3)
Traceback (most recent call last):
  ...
TypeError: add expected 2 arguments, got 1

    NameError

    • Cause: 变量没有被赋值给任何东西，或者它不存在。这包括函数名称。

    • Example: 

      File "file name", line number
  y = x + 3
NameError: global name 'x' is not defined

    • Solution: 在使用变量之前，请确保您正在初始化变量（即将变量赋值）。

    • Notes: 错误消息显示“全局名称”的原因是因为Python将从函数的局部框架开始搜索变量。如果在那里找不到变量，Python将继续搜索父帧，直到到达全局帧。如果仍然找不到变量，Python会抛出错误。

    IndexError

    • Cause: 试图用超过序列大小的数字索引序列（例如元组，列表，字符串）。

    • Example: 

      File "file name", line number
  x[100]
IndexError: tuple index out of range

    • Solution: 确保索引在序列的范围内。如果你使用一个变量作为索引（例如 seq[x] ），请确保该变量被分配给一个正确的索引。

    Common Bugs 常见错误

    Spelling 拼写

    Python是大小写敏感的。变量 hello 与 Hello 或 hello 或 helo 不同。这通常会显示为 NameError ，但有时拼写错误的变量实际上已经定义。在这种情况下，很难找到错误，而且发现这只是一个拼写错误也永远不会令人满意。

    Missing Parentheses 缺少括号

    一个常见的错误是省略了右括号。这将显示为 SyntaxError 。下面是一个例子：

    def fun():
    return foo(bar()   # missing a parenthesis here

fun()

    Python将引发一个 SyntaxError ，但会指向缺少括号后的行：

    File "file name", line "number"
    fun()
      ^
SyntaxError: invalid syntax

    一般来说，如果Python将 SyntaxError 指向一个看似正确的行，你可能在某处忘记了括号。

    Missing close quotes 缺少右引号

    这与前一个bug类似，但更容易捕获。Python实际上会告诉你缺少引号的那一行：

    File "file name", line "number"
  return 'hi
           ^
SyntaxError: EOL while scanning string literal

    EOL 代表“End of Line”。“

    = vs==

    单等号 = 用于赋值;双等号 == 用于测试等效性。这种形式最常见的错误是：

    if x = 3:

    Infinite Loops 死循环

    无限循环通常由条件永远不变的 while 循环引起。举例来说，请注意：

    i = 0
while i < 10:
    print(i)

    有时您可能会增加错误的计数器：

    i, n = 0, 0
while i < 10:
    print(i)
    n += 1

    Off-by-one errors 差一点的错误

    有时候 while 循环或递归函数可能会不在我们预期的地方停止。在这里，最好遍历迭代/递归，看看循环在哪个数字处停止。

    下面是一个示例，展示了一个常见的off-by-one错误：

    # 计算列表中的元素总和
def calculate_sum(numbers):
    total = 0
    index = 1  # 错误的索引起始点

    while index <= len(numbers):  # 错误的停止条件
        total += numbers[index]  # 错误的索引访问
        index += 1

    return total

numbers = [1, 2, 3, 4, 5]
print(calculate_sum(numbers))

    在这个例子中，calculate_sum函数的目标是计算给定列表中元素的总和。然而，存在一个off-by-one错误导致结果不正确。

    错误的地方包括：

    • 索引起始点：变量index的初始值设置为1，而不是0。在Python中，列表的索引是从0开始的。
    • 停止条件：循环的停止条件是index <= len(numbers)，这会导致循环执行了多一次。应该使用index < len(numbers)才是正确的条件。
    • 索引访问：在计算总和时，应该使用numbers[index - 1]来访问列表中的元素，因为索引是从0开始的。

    这不会引起解释器报错，但是会得出我们不想要的结果。
