    JWT工作流程

    Java > Spring 后端 安全框架 JWT
    发布时间 : 2023-08-15 15:59
    字数:947 阅读 :
    评论:

    ​​

    1. 我们发送了一个对正在运行的后端系统的 HTTP 请求

      值得一提的是，Spring 中执行的第一个应用程序是过滤器。所以无论何时何地创建一个过滤器，只需注意并记住它将是我们的应用程序中执行的第一件事

    2. 所以在这种情况下，首先要执行的是我们的 JWT 身份验证过滤器

      这只是一次请求过滤器并具有验证和检查我们拥有的令牌或 JWT 令牌的所有内容的作用

    3. 现在让我们开始这个过程（ JWT 身份验证过滤​ ）

      首先要做的就是发生这种情况时，我们将进行内部检查，以检查我们是否有 JWT 令牌。因此，如果令牌丢失，客户端将发送403响应返回给我们，原因就是缺少 JWT。

      如果我们拥有 JWT 令牌，我们将启动过程或验证过程。所以这个验证过程将以这种方式开始。过滤器就在内部执行

    4. 首先会使用用户名（电子邮件）尝试从数据库调取用户详细信息

      我们将基于用户电子邮件设置为我们将提取的声明或令牌主题此 JWT 身份验证过滤器将检查 JWT 令牌，提取用户名或电子邮件，或者我们将其称为Subject​。当我们谈论 JWT 令牌时，它将使用该电子邮件来从我们的数据库中获取用户详细信息（UserDetails）。然后一旦获取了用户，我们从数据库中得到了响应，并且响应可以使得无论哪种方式都可以是现有用户或非现有用户。一旦我们收到 JWT 身份验证的响应，过滤器将进行一些检查，即用户是否不存在，否则后端服务器也会向用户发送 403。如果一切都很好，我们从数据库中获取用户

    5. 然后我们将开始验证过程（ Validate JWT ）

      这个验证过程，因为此 JWT 令牌是为用户特定生成的，因此我们要根据用户验证此令牌（也就是 JWT令牌适合用户相绑定的，JWT令牌被持有）

      这里我们有这个验证 JWT 流程或机制，将尝试调用 JWT 服务。这个 JWT 服务将需要作为参数的用户本身（即 UserDetails ）以及令牌或字符串令牌或 JWT 令牌，这个可以随意命名

    6. 之后执行这个验证过程，这里有两种情况

      设想第一个情况：令牌无效。例如，令牌已过期或令牌不适用于该特定用户。那么我们要做的就是发回 403给我们的客户。

      第二个情况：令牌有效。我们会更新SecurityContextHolder​并设置此连接用户。因为当我们从数据库获取用户详细信息时，我们将能够设置SecurityContextHolder​

      所以我们会告诉服务器或者过滤器链的其余部分表明该用户现已通过身份验证我们将更新身份验证管理器。所以每当我们检查该用户是否通过该请求的身份验证时，答案

      是肯定的。一旦SecurityContextHolder​更新后，会自动发送请求，它将被发送到DispatcherServlet​。并从DispatcherServlet​，它将直接发送到 Controller​

    我们将执行我们需要执行的所有操作，例如，调用服务，访问数据库等等，所以。然后我们会发回回复。例如，它可以是 JWT，它可以是 HTTP 200，或者任何进程将在此控制器中执行。所以这就是如何JWT 身份验证机制有效。
